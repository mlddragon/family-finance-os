# Report And Monthly Close Artifacts v1

This document defines the proposed v1 report and monthly close artifact structure for Dillon Finances. It is a planning artifact only. It does not create report code, create schema, add dependencies, generate artifacts, or migrate financial data.

## Status

- Proposed for owner review.
- App implementation has not started.
- No reports or monthly close files have been generated.
- Exact output schemas remain deferred to implementation planning.

## Recommendation

Strong recommendation: use database-backed report runs with durable, human-readable file artifacts, and treat monthly close as an immutable bundle once finalized. This keeps the app queryable while preserving inspectable evidence for Mason, Codex, ChatGPT/OpenAI analysis, and future advisor workflows. Reports can be provisional and regenerated; finalized monthly close bundles should be versioned rather than silently edited.

Serious alternatives considered:

- Database-only reports: simpler for the app, but too opaque for audit, backup, and advisor workflows.
- File-only reports: easy to inspect, but risks recreating CSV-as-state and makes artifact discovery/versioning fragile.
- Finalize in place by overwriting monthly close files: convenient, but unsafe because it erases historical reporting context.

## Artifact Storage Boundary

Artifacts are stored under `DATA_ROOT`, outside git.

Proposed folders:

```text
DATA_ROOT/
  reports/
    runs/
      {report_run_id}/
  monthly_close/
    YYYY-MM/
      draft/
      final/
      revisions/
  exports/
    advisor/
    audit/
```

Rules:

- Generated reports and close bundles are never committed to git.
- SQLite stores artifact metadata, job links, hashes, source inputs, and readiness state.
- Filesystem artifacts remain inspectable and portable.
- CSV, JSON, and Markdown are artifact/export formats, not active product state.

## Artifact Registry Contract

Every generated artifact should have a SQLite artifact record.

Artifact metadata should capture:

- Artifact id.
- Artifact type.
- Title.
- Description.
- Filesystem path.
- File hash.
- Byte size.
- Created timestamp.
- Producing job id.
- Report run id or close id when applicable.
- Source input ids or dependency ids.
- Validation state snapshot id.
- Retention category.
- Sensitive-data classification.
- Provisional/final status.

Artifact types:

- Validation summary.
- Validation findings table.
- Cashflow summary.
- Category spending summary.
- Review backlog summary.
- Top merchants/sources summary.
- Monthly close memo.
- Reviewed transaction export.
- Decision event export.
- Settings snapshot.
- Advisor export.
- Job log.

## Report Run Contract

Reports should be generated by backend jobs from validated SQLite-backed state.

Each report run should capture:

- Report run id.
- Report type.
- Period.
- Created timestamp.
- Actor.
- Producing job id.
- Input import batch ids.
- Input decision event range or snapshot id.
- Input settings snapshot id.
- Validation state snapshot id.
- Provisional/final status.
- Artifact ids produced.
- Error summary if failed.

Rules:

- Reports are generated as jobs, not on every page load.
- Reports must carry validation and freshness status.
- Reports can be generated provisionally when required sources are missing or stale.
- Reports must not present provisional data as final.
- Report reruns create new report run records and artifacts.

## v1 Report Set

### Import And Validation Summary

Purpose:

- Show import coverage, freshness, validation health, and duplicate/schema/date/amount risks.

Primary format:

- Markdown summary for owner review.
- CSV/JSON findings table for audit and UI drilldown.

Minimum content:

- Sources included.
- Required sources missing.
- Latest successful import per source.
- Latest transaction date per source.
- Fresh/stale state by source.
- Files included.
- File hashes.
- Row counts by file/source.
- Transaction date min/max by source.
- Validation counts by severity.
- Open blocking findings.
- Open warning findings.
- Accepted warnings.
- Duplicate/overlap summary.

### Cashflow Summary

Purpose:

- Show month-level income, spending, transfers, and net cashflow.

Primary format:

- CSV/JSON table.
- Markdown narrative when included in monthly close memo.

Minimum dimensions:

- Month.
- Income.
- Spending.
- Transfers.
- Net cashflow.
- Source coverage status.
- Validation/provisional status.

Rules:

- Ledger transactions are the source of truth.
- Vendor/detail sources do not create cashflow rows.
- Transfers should be separated from spending/income when identifiable.

### Category Spending Summary

Purpose:

- Show spending by category/subcategory with review exposure.

Primary format:

- CSV/JSON table.

Minimum dimensions:

- Month.
- Category.
- Subcategory.
- Spending amount.
- Transaction count.
- Reviewed amount.
- Unreviewed amount.
- Review exposure percentage.
- Provisional status.

Rules:

- Category totals must preserve ledger cashflow integrity.
- Reports must expose unreviewed exposure instead of hiding uncertainty.

### Review Backlog Summary

Purpose:

- Show review work that blocks confidence or should be prioritized.

Primary format:

- CSV/JSON table.
- Markdown summary for monthly close.

Minimum dimensions:

- Review reason.
- Count.
- Dollar exposure.
- Source.
- Category/subcategory when available.
- Blocking/warning/provisional impact.

### Top Merchants/Sources

Purpose:

- Show high-impact merchants and sources for prioritization.

Primary format:

- CSV/JSON table.

Minimum dimensions:

- Merchant/source.
- Category.
- Amount.
- Transaction count.
- Review exposure.
- Provisional status.

### Monthly Close Memo

Purpose:

- Provide the narrative owner/advisor summary for the close period.

Primary format:

- Markdown.

Minimum sections:

- Close status.
- What changed.
- What is safe to trust.
- What remains provisional.
- Open blockers.
- Review exposure.
- Recommended next actions.
- Source coverage and freshness.
- Links/references to generated artifacts.

Rules:

- Memo must cite or reference validation and report artifacts.
- Memo must clearly label draft/provisional/final state.

## Monthly Close Lifecycle

Recommended lifecycle:

```text
not started
  -> draft generated
  -> blocked / ready for final review
  -> finalized
  -> revised, if correction is needed later
```

### Draft Close

Allowed when:

- At least one required source has accepted data for the period.
- Import validation has run.
- Open blocking findings are visible.

Rules:

- Draft close can be provisional.
- Draft close can include advisor-ready draft exports only when clearly labeled.
- Draft close can be regenerated.

### Final Close

Allowed only when:

- All required sources are present for the close period.
- Required sources are not stale for the close period.
- Blocking validation findings are resolved.
- Duplicate ambiguity that could affect totals is resolved.
- Report artifacts have been generated from the accepted state.
- Settings snapshot and decision event export are captured.

Rules:

- Final close is immutable.
- Final close should not be overwritten.
- Corrections create a revised close version.

### Revised Close

Required when:

- A later correction changes close-period facts, decisions, settings, or report logic.

Rules:

- Revision keeps the prior final close intact.
- Revision records supersedes/superseded-by links.
- Revision includes a reason/note.

## Monthly Close Bundle Structure

Recommended final bundle:

```text
DATA_ROOT/monthly_close/YYYY-MM/final/
  manifest.json
  validation/
    validation_summary.md
    validation_findings.csv
  reports/
    cashflow_summary.csv
    category_spending_summary.csv
    review_backlog_summary.csv
    top_merchants_sources.csv
  memo/
    monthly_close_memo.md
  exports/
    reviewed_transactions.csv
    decision_events.csv
    settings_snapshot.json
  advisor_export/
    advisor_summary.md
    advisor_tables.json
```

Draft bundles may use the same structure under:

```text
DATA_ROOT/monthly_close/YYYY-MM/draft/
```

Revisions should use:

```text
DATA_ROOT/monthly_close/YYYY-MM/revisions/{revision_id}/
```

## Manifest Contract

Each monthly close bundle should include `manifest.json`.

Minimum manifest fields:

- Close id.
- Month.
- Status: draft, final, or revised.
- Created timestamp.
- Finalized timestamp when applicable.
- Actor.
- Source import batch ids.
- Report run ids.
- Artifact ids.
- Validation state summary.
- Settings snapshot id.
- Decision export id.
- Provisional reason list.
- Supersedes close id when applicable.
- File inventory with hashes and byte sizes.

Rules:

- Manifest should be machine-readable.
- Manifest should not include raw credentials or unnecessary full account identifiers.
- Manifest is the first file the UI/advisor flow should inspect when loading a bundle.

## Advisor Export Contract

Advisor export is allowed for v1 as an explicit owner-directed export.

Recommended contents:

- `advisor_summary.md`: concise narrative with source coverage, validation state, provisional labels, and next actions.
- `advisor_tables.json`: structured tables needed for external analysis.

Rules:

- Advisor export must include validation status and source freshness.
- Advisor export must label provisional data.
- Advisor export should prefer reviewed/current derived state over raw imported rows.
- Raw transaction detail can be included only when explicitly generated by owner action for advisor use.
- No export should include credentials, secrets, browser session data, or unnecessary full account identifiers.

## Retention

Recommended default:

- Keep raw imports, monthly close bundles, report artifacts, and export bundles indefinitely until an owner-approved archive/purge policy exists.
- Intermediate draft report runs can be replaceable before final close.
- Final close bundles and revisions are retained.

## UI Expectations

The Reports / Monthly Close UI should show:

- Report readiness before report content.
- Provisional/final status.
- Required source coverage.
- Blocking validation count.
- Unreviewed exposure.
- Settings snapshot availability.
- Generated artifact list.
- Monthly close status.
- Advisor export status.

The UI should not infer artifact meaning by scanning folders alone. It should use artifact registry and close manifest metadata.

## Explicit Non-Goals

- No report implementation.
- No database schema.
- No generated report files.
- No generated monthly close bundle.
- No financial data artifacts.
- No AI API integration.
- No vendor item reports in v1.
- No budget target variance in v1.
- No net worth or retirement reports in v1.
